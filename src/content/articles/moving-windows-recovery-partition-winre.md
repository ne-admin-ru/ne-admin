Иногда расширить диск `C:` в Windows мешает маленький recovery-раздел, который стоит сразу после системного раздела. В виртуалках это особенно заметно: диск увеличили, свободное место появилось, а расширить основной раздел нельзя, потому что между ними стоит Windows Recovery Environment.

На первый взгляд хочется просто удалить этот раздел и забыть. Но это плохая привычка: WinRE нужен для восстановления системы, сброса, части сценариев обновлений и нормальной диагностики после сбоев.

Наткнулся на хороший англоязычный разбор theDXT про перенос recovery-раздела без сторонних утилит. Ниже фиксирую суть своими словами, чтобы не искать заново, когда снова попадется такая схема диска.

## Почему тема всплыла

Microsoft изменила обслуживание WinRE: среда восстановления стала обновляться вместе с ежемесячными накопительными обновлениями. Если recovery-раздел слишком маленький, обновление может не установиться. В KB5028997 Microsoft отдельно описывает ручное изменение размера раздела WinRE.

То есть это не косметика. Маленький или неудобно расположенный recovery-раздел может мешать и расширению диска, и обслуживанию Windows.

## Главная мысль

Раздел восстановления нельзя переносить как обычную папку. Правильная логика такая:

```text
проверить WinRE
-> отключить WinRE
-> освободить место
-> пересоздать recovery-раздел в нужном месте
-> выставить правильный тип раздела и атрибуты
-> включить WinRE обратно
-> проверить результат
```

Ключевой момент - сначала выполнить `reagentc /disable`. После этого Windows сохраняет образ восстановления `Winre.wim` в системный каталог, и старый recovery-раздел можно пересоздать более контролируемо.

## Перед началом

Это операции с разделами диска. Перед такими действиями нужен бэкап, а для виртуалки - снапшот. Не стоит копировать команды вслепую: номера дисков и разделов почти всегда отличаются.

Минимальная проверка перед работой:

```powershell
reagentc /info
```

В выводе надо посмотреть статус WinRE и путь вида:

```text
\\?\GLOBALROOT\device\harddisk0\partition4\Recovery\WindowsRE
```

По нему видно, на каком диске и разделе сейчас живет WinRE.

## Общий порядок

Сначала отключаем WinRE:

```powershell
reagentc /disable
```

Дальше через `diskpart` проверяем диск и разделы:

```text
diskpart
list disk
select disk 0
list partition
```

`disk 0` здесь только пример. Нужно выбрать свой диск, сверившись с выводом `reagentc /info` и реальной разметкой.

После этого выбирается старый recovery-раздел и удаляется:

```text
select partition 4
delete partition override
```

Опять же, `partition 4` - пример. Ошибка с номером раздела может удалить не то, что нужно.

Затем расширяется основной раздел, но в конце диска оставляется место под новый recovery-раздел. В статье theDXT для примера оставлен 1 ГБ. Microsoft в KB5028997 описывает сценарий добавления 250 МБ к существующему разделу, но на практике размер лучше выбирать с запасом и с учетом текущей схемы Windows.

После создания нового раздела важно выставить правильный тип.

Для GPT-диска:

```text
set id=de94bba4-06d1-4d40-a16a-bfd50179d6ac
gpt attributes=0x8000000000000001
```

Для MBR-диска:

```text
set id=27
```

Понять GPT или MBR можно в `diskpart` через `list disk`: если в колонке `Gpt` есть звездочка, диск GPT.

После этого включаем WinRE обратно:

```powershell
reagentc /enable
reagentc /info
```

В финале важно убедиться, что WinRE снова включен и указывает на новый recovery-раздел.

## Что мне нравится в этом подходе

Он не требует GParted, загрузочных ISO и платных менеджеров разделов. Для серверной или облачной виртуалки это удобно: меньше ручной возни, меньше простоев, все делается штатными средствами Windows.

Но подход все равно остается рискованным. Это не команда из разряда "попробовать наугад". Сначала бэкап, потом проверка схемы диска, потом аккуратные действия.

## Итог

Если recovery-раздел мешает расширить `C:`, его не обязательно навсегда удалять. Более правильный путь - временно отключить WinRE, пересоздать раздел в новом месте, выставить recovery-атрибуты и включить WinRE обратно.

Главное - не делать это механически. Проверить `reagentc /info`, понять GPT или MBR, сверить номера разделов, иметь бэкап и только потом трогать разметку диска.
