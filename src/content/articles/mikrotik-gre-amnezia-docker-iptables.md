Интересный случай из практики: был VPS, офисный MikroTik и задача выпустить часть офисного трафика в интернет через VPS. На первый взгляд схема простая: поднимаем GRE между VPS и MikroTik, на роутере отправляем нужную подсеть в туннель, на VPS делаем форвардинг и masquerade наружу.

Но рядом на том же VPS жила Amnezia. А Amnezia - это Docker, свои интерфейсы, свои цепочки и свой взгляд на `iptables`. В итоге схема работала, потом после изменений или перезапуска VPN-контейнеров могла перестать работать, хотя сам GRE при этом выглядел живым.

## Схема

Упрощенно было так:

```text
офисная сеть
  -> MikroTik
  -> GRE-туннель
  -> VPS
  -> NAT/MASQUERADE
  -> интернет
```

На VPS поднимался GRE-интерфейс, например `gre1`, с адресом из маленькой point-to-point подсети `/30`. На MikroTik создавался ответный GRE-интерфейс. После этого офисная LAN могла маршрутизироваться через адрес VPS внутри GRE.

На VPS были нужны три вещи:

```text
- разрешить входящий GRE, то есть IP protocol 47;
- включить IPv4 forwarding;
- разрешить FORWARD из GRE в WAN и обратно для established/related;
- сделать MASQUERADE офисной подсети во внешний интерфейс VPS.
```

В нормальном состоянии этого достаточно: клиент из офиса идет на MikroTik, MikroTik отправляет пакет в GRE, VPS принимает его, NAT'ит и выпускает в интернет со своего внешнего IP.

## Где стало интересно

Проблема была не в GRE как таковом. Туннель мог подниматься, интерфейс существовал, адреса были назначены. Но трафик офисной подсети переставал нормально проходить через VPS.

После разбора стало понятно, что рядом вмешивается Docker/Amnezia. При запуске и обновлениях контейнерной части Docker меняет правила `iptables`, добавляет свои цепочки и может менять порядок обработки. В таких схемах порядок правил критичен: если нужный `FORWARD` или `MASQUERADE` оказался ниже Docker-цепочек или перекрыт более ранним правилом, трафик уже не матчится так, как ожидалось.

Снаружи это выглядит неприятно: GRE вроде есть, пинг до пира может оживать после ручного действия, но пользователи из офисной сети не выходят через VPS.

## Что оказалось важным

Первое - правила надо вставлять в начало цепочек, а не просто добавлять в конец.

Например, для входящего GRE логика такая:

```bash
iptables -I INPUT 1 -p gre -s <MIKROTIK_PUBLIC_IP> -j ACCEPT
```

Для форвардинга офисной сети:

```bash
iptables -I FORWARD 1 -i gre1 -s <OFFICE_LAN> -o <WAN_IF> -j ACCEPT
iptables -I FORWARD 2 -i <WAN_IF> -o gre1 -d <OFFICE_LAN> -m state --state ESTABLISHED,RELATED -j ACCEPT
```

Для NAT:

```bash
iptables -t nat -I POSTROUTING 1 -s <OFFICE_LAN> -o <WAN_IF> -j MASQUERADE
```

Второе - перед вставкой надо удалять старые копии своих правил. Иначе после каждого перезапуска сервиса можно накопить одинаковые строки в `iptables`.

Идея простая:

```bash
while iptables -D FORWARD ... 2>/dev/null; do :; done
iptables -I FORWARD 1 ...
```

Так сервис становится идемпотентным: его можно перезапускать после Docker/Amnezia, после ребута или после ручных правок, не боясь раздуть таблицу правил.

## Отдельно про rp_filter

В GRE-схемах с асимметричной маршрутизацией может мешать reverse path filtering. Поэтому для туннельного интерфейса и иногда глобально приходилось отключать строгую проверку обратного пути:

```bash
sysctl -w net.ipv4.conf.gre1.rp_filter=0
sysctl -w net.ipv4.conf.all.rp_filter=0
```

Это не универсальная рекомендация "выключать везде", а именно типичная проверка для случаев, когда пакет приходит по одному интерфейсу, а обратный маршрут с точки зрения ядра выглядит неоднозначно.

## Как сделали устойчиво

В итоге на VPS был вынесен отдельный systemd-сервис, который каждый раз пересоздает GRE-интерфейс и заново применяет нужные правила firewall/NAT.

Логика сервиса:

```text
удалить старый gre1, если есть
-> создать GRE до MikroTik
-> назначить адрес /30
-> включить ip_forward
-> ослабить rp_filter для GRE-сценария
-> удалить старые копии своих iptables-правил
-> вставить свежие правила в начало цепочек
-> применить дополнительные маршруты, если они есть
```

Дополнительно был добавлен health-check через systemd timer. Он каждые несколько секунд проверяет, есть ли `gre1` и пингуется ли GRE-пир. Если нет - запускает основной скрипт восстановления.

```text
systemd timer
  -> ping peer через gre1
  -> если сбой, пересоздать туннель и правила
```

Это не заменяет нормальный мониторинг, но для небольшого рабочего кейса дает понятный самовосстанавливающийся контур.

## Что делалось на MikroTik

На стороне MikroTik схема была стандартная:

```rsc
/interface gre
add name=gre-to-vps local-address=<MIKROTIK_PUBLIC_IP> remote-address=<VPS_PUBLIC_IP>

/ip address
add address=<MIKROTIK_GRE_IP>/30 interface=gre-to-vps
```

Дальше нужен маршрут для нужной офисной подсети или нужных направлений через GRE-пир на VPS. Если через VPS должен идти весь трафик выбранных клиентов, это лучше делать аккуратно: отдельной address-list, policy routing или отдельной таблицей маршрутизации, чтобы не отправить в туннель весь офис неожиданно.

## Как проверять

Минимальный набор проверок на VPS:

```bash
ip link show gre1
ip addr show gre1
ip route
iptables -S FORWARD
iptables -t nat -S POSTROUTING
ping -I gre1 <MIKROTIK_GRE_IP>
```

На MikroTik:

```rsc
/interface gre print
/ip address print
/ip route print
/ping <VPS_GRE_IP>
```

С клиента в офисной сети полезно проверить внешний адрес:

```bash
curl -4 ifconfig.me
```

Если все собрано правильно, внешний IP должен быть IP VPS, а не офисного канала.

## Итог

Главный вывод: когда на одном VPS живут VPN-контейнеры, Docker и ручная маршрутизация через GRE, нельзя считать `iptables` статичной картинкой. Docker и Amnezia могут менять порядок цепочек, а для GRE/NAT порядок часто решает все.

Рабочее решение - не просто "один раз добавить правила", а сделать воспроизводимый скрипт, который:

```text
- пересоздает GRE;
- удаляет свои старые правила;
- вставляет новые правила в начало цепочек;
- учитывает rp_filter;
- запускается через systemd;
- имеет простой health-check.
```

Такой подход превращает хрупкую ручную настройку в нормальный эксплуатационный сценарий: если Amnezia или Docker снова переставят правила, сервис можно перезапустить и вернуть маршрут в рабочее состояние.
